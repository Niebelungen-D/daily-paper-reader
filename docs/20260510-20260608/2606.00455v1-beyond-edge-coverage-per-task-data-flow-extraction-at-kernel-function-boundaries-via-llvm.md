## 摘要
基于覆盖的内核模糊测试工具（如syzkaller）仅依赖边覆盖（trace-pc）作为反馈信号。这种上下文盲方法无法区分仅参数值不同的执行路径。例如，两次copy_from_user()调用使用不同size参数会命中相同基本块，但具有截然不同的安全影响。我们提出BOUNDARY FLOW，一个基于LLVM的插桩框架，通过函数参数和返回值的数 据 流 提 取 扩展Linux KCOV。编译器传递（-fsanitize-coverage=dataflow-args, dataflow-ret）发出轻量回调，在函数入口捕获结构化元组<PC, arg_idx, arg_size, ptr, offsets[]>，在返回时捕获<PC, ret_size, ptr, offsets[]>。复合类型通过DWARF DICompositeType元数据自动分解，无需源代码注释。一个独立的内核设备（/sys/kernel/debug/kcov_dataflow）提供无锁的每任务环形缓冲区，不干扰现有KCOV或syzkaller基础设施。我们证明了双重用途：模糊测试器获得状态感知反馈，用于指导进入值依赖状态转移的变异；安全分析人员获得确定性参数记录，用于根因分析而无需printk或kprobe开销。一个后编译流水线（rustc, opt, llc）支持Rust内核模块插桩而无需修改rustc，这是捕获Rust函数参数唯一的运行时方法（因为drgn/vmcore在-O2 DWARF省略下失败）。在五个漏洞类别（OOB、UAF、double-free、10层深度链传播、Rust FFI、Rust for Linux Modules）上评估，插桩路径开销低于3%。

## Abstract
Coverage-guided kernel fuzzers such as syzkaller rely on edge coverage (trace-pc) as their sole feedback signal. This context-blind approach cannot distinguish execution paths that differ only in argument values. for example, two invocations of copy_from_user() with different size parameters hit identical basic blocks yet have vastly different security implications. We present BOUNDARY FLOW, an LLVM-based instrumentation framework that extends Linux KCOV with data-flow extraction of function arguments and return values. A compiler pass (-fsanitize-coverage=dataflow-args, dataflow-ret) emits lightweight callbacks capturing a structured tuple <PC, arg_idx, arg_size, ptr, offsets[]> at function entry and <PC, ret_size, ptr, offsets[]> at return. Composite types are automatically decomposed via DWARF DICompositeType metadata with zero source annotation. A separate kernel device(/sys/kernel/debug/kcov_dataflow) provides lock-free per-task ring buffers with no inter ference to existing KCOV or syzkaller infrastructure. We demonstrate dual utility: fuzzers gain state-aware feedback for mutation guidance into value-dependent state transitions, and security analysts obtain deterministic argument records for root-cause analysis without printk or kprobe overhead. A post-compilation pipeline (rustc, opt, llc) enables Rust kernel module instrumentation without modifying rustc, the only runtime method for capturing Rust function arguments given that drgn/vmcore fails under-O2 DWARF elision. Evaluated on five vulnerability classes (OOB, UAF, double-free, 10 deep chain propagation, Rust FFI, Rust for Linux Modules) with <3% overhead on instrumented paths.

### 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：现有覆盖引导的内核模糊测试（如 syzkaller）仅依赖边覆盖（trace-pc）作为反馈信号，无法区分仅参数值不同的执行路径。例如，两次 `copy_from_user()` 调用使用不同 `size` 参数会命中相同基本块，但安全影响截然不同。这种“语义鸿沟”导致模糊测试在值依赖的状态转移子系统中（如 ksmbd、io_uring、binder）容易饱和，难以发现深层次漏洞。
- **研究动机**：函数边界是天然的观察点，在此处捕获参数和返回值的数据流既廉价又信息丰富。作者旨在通过编译时插桩自动提取数据流上下文，为模糊测试提供状态感知反馈，同时为安全分析人员提供确定性参数记录，无需源代码注释或额外运行时开销。
- **整体含义**：提出 BOUNDARY FLOW 框架，通过扩展 LLVM SanitizerCoverage 和 Linux KCOV，在不干扰现有基础设施的前提下，以极低开销（插桩路径 <3%）突破边覆盖的局限性，同时支持 Rust 内核模块插桩（这是唯一能在 -O2 DWARF 省略下捕获 Rust 函数参数的方法）。

### 2. 方法论：核心思想、关键技术细节

- **核心思想**：在函数边界（入口和返回）自动捕获参数和返回值的结构化数据流元组，通过编译器插桩和内核无锁环形缓冲区传输，实现零源代码注释的数据流感知。
- **关键技术细节**：
  1. **LLVM IR 边界插桩**：在 SanitizerCoverage 框架中新增两个 pass：
     - **参数捕获**（`InjectTraceForArgs`）：遍历函数参数，通过 DISubprogram 和 DICompositeType 元数据递归解析 struct/union 的字段偏移和大小，生成编译时常量数组（[类型哈希, 偏移0, 大小0, …]）；在函数入口插入回调 `__sanitizer_cov_trace_args(PC, arg_idx, arg_size, ptr, offsets_ptr, num_fields)`。标量参数被溢出到 alloca 以统一指针接口。
     - **返回值捕获**（`InjectTraceForRet`）：使用 EscapeEnumerator 定位所有 ReturnInst，在返回前插入回调 `__sanitizer_cov_trace_ret(PC, ret_size, ptr, offsets_ptr, num_fields)`。
  2. **锁无关的每任务环形缓冲区**：在 `task_struct` 中添加三个字段（`kcov_df_area`, `kcov_df_size`, `kcov_df_seq`）。回调通过原子加（`atomic_add`）预留槽位，无阻塞、无分配、无自旋锁。使用 `copy_from_kernel_nofault()` 安全读取指针指向的内存（处理 KASAN 毒化、已释放 slab、NULL、ERR_PTR 等）。记录格式为 3 字头 + N 个字段值，包含 type_and_seq（入口/返回标记+序列号）、PC、meta（参数索引/返回标记、大小、指针低48位）。
  3. **安全机制**：回调标记为 `notrace`、`__no_sanitize_coverage`、`noinline`；通过 `in_task()` 守卫拒绝中断和 NMI 上下文调用，避免竞争和死锁。
  4. **独立设备接口**：通过 `/sys/kernel/debug/kcov_dataflow` 设备提供独立的 ioctl（`KCOV_DF_INIT_TRACE`、`KCOV_DF_ENABLE`、`KCOV_DF_DISABLE`），与现有 KCOV 完全解耦。
  5. **Rust 模块支持**：通过后编译流水线（`rustc --emit=llvm-ir` → `opt` 应用自定义 pass → `llc` 编译为目标文件），利用 LLVM IR 语言无关性，避免修改 rustc 工具链。支持 `#[repr(C)]` 结构体的字段展开。
  6. **动态结构体捕获**：支持堆上动态分配的结构体，回调在运行时通过 `ptr + offset[i]` 计算字段地址并读取，实现一级指针解引用（不追踪结构体内嵌套指针）。

### 6. 主要结论与发现

- **正确性**：KCOV-DATAFLOW 成功捕获所有 5 个漏洞类别的入口/返回值，与 printk ground truth 完全匹配。
- **信息增益**：与 KASAN 和 drgn 相比，KCOV-DATAFLOW 能提供：哪个参数被破坏、破坏前后值、跨函数传播路径；对于 Rust 模块，是唯一能在 -O2 下捕获参数的方法。
- **性能**：微基准中单个回调开销约 27 ns，记录开启后总迭代延迟增加 8.3%（16.3 μs vs 15.0 μs）；全局插桩（全内核开启但未记录）导致系统调用延迟增加 133%（受 per-function 常数数组和 prologue 代码影响），但仍低于 KASAN 或 KMSAN。
- **真实世界发现**：在 Android Binder 和 Rust 核心子系统中发现 60 个测试用例，其中 4 个高严重度缺陷（错误掩码、binder 线程耗尽等），8 个中严重度（缺乏验证、语义问题），以及多个正确的边界处理确认。
- **Rust 支持**：成功通过后编译流水线实现 Rust 模块插桩，正确捕获 8 参数混合类型函数（含结构体指针）。
- **边界面合同验证**：提出将 KCOV-DATAFLOW 输出作为 post-mortem 形式化验证基础，可离线检查谓词/时序逻辑，自动发现契约违反。

### 7. 优点

- **零源代码修改**：完全自动化的编译器 pass，无需开发者手动注释或添加 tracepoint。
- **极低开销**：记录路径平均仅 8-9% 延迟增加，全局全插桩但未记录时系统调用延迟虽增 133%，但仍在可接受范围（低于 KASAN），且可选择性启用模块级插桩。
- **跨语言支持**：通过 LLVM IR 后编译流水线同时支持 C 和 Rust，无需修改 rustc，解决了 Rust 内核模块调试的痛点。
- **安全设计**：锁无关环形缓冲区、中断/NMI 拒绝、`copy_from_kernel_nofault()` 安全读取、错误回退，确保在模糊测试压力下不引入新安全问题。
- **独立基础设施**：与现有 KCOV 和 syzkaller 完全解耦，可单独启用，兼容现有工作流。
- **丰富的信息输出**：提供结构化元组（PC、参数索引、类型哈希、字段偏移、字段值），支持类型去重和跨函数传播追踪，便于自动化分析。

### 8. 不足与局限

- **依赖调试信息**：完整结构体字段展开需要 `-g`（CONFIG_DEBUG_INFO），否则只能捕获标量参数（对于内核模糊测试，KASAN 已要求调试信息，实际限制较小）。
- **Rust 后编译流水线未集成**：当前需手动调用 wrapper 脚本，尚未融入 Kbuild 自动构建；作者提议未来在 rustc 中原生支持 `-Zsanitizer-coverage=dataflow-args`。
- **缓冲区溢出丢包**：当产生事件超过缓冲区容量时静默丢弃，虽为设计选择，但可能丢失关键点。对于快速连续调用的函数，需增大缓冲区或用户空间及时消费。
- **中断/NMI 上下文不可见**：`in_task()` 守卫丢弃所有非进程上下文回调，导致 IRQ/NMI 触发的漏洞无法捕获（但这些漏洞通常无法通过 syscall 路径的模糊测试触发，实际影响有限）。
- **仅一级指针解引用**：不递归解析结构体内的指针字段，可能遗漏深层数据流（如 `char *buf` 指向的内容），但指针值本身已有信息量（NULL、KASAN 毒化等）。
- **实验局限**：未进行与 Angora、GREYONE 等数据流模糊测试工具的直接对比（因它们主要面向用户态）；未在长时间（如多日）syzkaller 运行中对比发现效率提升；性能测试在虚拟化环境下进行，真实硬件效果可能略有不同。
- **上游集成待推进**：未提及实际提交到 LLVM 和 Linux 主线的情况，尚属研究原型。
