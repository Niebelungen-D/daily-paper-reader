## 摘要
大型语言模型（LLM）现已支持自动化软件安全任务，包括漏洞发现和概念验证（PoC）生成。现有基准测试未能忠实评估LLM在真实漏洞狩猎场景中的表现，因为它们依赖于模糊测试工具、特定目标描述或漏洞复现任务。我们提出了SEC-bench Pro，一个用于衡量智能体在关键高复杂度软件系统上漏洞狩猎能力的基准测试。本工作通过三阶段流水线（漏洞收集、环境重建和基于oracle的验证）披露了带有具体PoC输入的报告，并将修复与可复现任务关联。我们以V8和SpiderMonkey中的183个已验证漏洞实例化了SEC-bench Pro，其中包括一个V8子集，其累积Google漏洞奖励计划奖金超过150万美元。这些实例涵盖了浏览器级和运行时级执行条件下的内存安全、沙箱、JIT和竞争条件漏洞。我们的评估表明，使用前沿模型的编码智能体在两个评估引擎上的成功率均低于40%。开源权重Kimi-K2.6基线在V8上达到11.7%，而最强前沿配置在V8上达到32.0%，在SpiderMonkey上达到38.8%。ClaudeCode和Codex解决了互补的实例集，它们的双智能体联合在V8上达到37.9%，在SpiderMonkey上达到48.8%。SEC-bench Pro为评估基于LLM的安全智能体提供了强大的环境，并揭示了长周期漏洞狩猎任务中的局限性。

## Abstract
Large language models (LLMs) now support automated software security tasks, including vulnerability discovery and proof-of-concept (PoC) generation. Existing benchmarks do not faithfully evaluate LLMs in real-world bug hunting scenarios because they rely on fuzzing harnesses, target-specific descriptions, or vulnerability-reproduction tasks. We present SEC-bench Pro, a benchmark for measuring agent bug hunting on critical, high-complexity software systems. This work discloses reports with concrete PoC inputs and links fixes into reproducible tasks through a three-phase pipeline for vulnerability collection, environment reconstruction, and oracle-based validation. We instantiate SEC-bench Pro with 183 validated vulnerabilities across V8 and SpiderMonkey, including a V8 subset with more than $1.5 million in cumulative Google Vulnerability Reward Program awards. These instances span memory-safety, sandbox, JIT, and race-condition bugs under browser-grade and runtime-grade execution conditions. Our evaluation shows that coding agents with frontier models remain below 40% success on both evaluated engines. The open-weight Kimi-K2.6 baseline reaches 11.7% on V8, while the strongest frontier configuration reaches 32.0% on V8 and 38.8% on SpiderMonkey. ClaudeCode and Codex solve complementary instance sets, and their two-agent union reaches 37.9% on V8 and 48.8% on SpiderMonkey. SEC-bench Pro provides robust environments for assessing LLM-based security agents and exposes limitations in long-horizon bug hunting tasks.

# SEC-bench Pro 论文总结

## 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**：当前大型语言模型（LLM）已能辅助软件安全任务，如漏洞发现和概念验证（PoC）生成。然而，现有安全基准（如CTF型、CVE型、依赖模糊测试工具的基准）存在三个主要缺口：
  - 依赖模糊测试工具（fuzz harness），评估的是“工具引导的输入变异能力”而非真正的代码审计；
  - 度量设计不合理，例如仅使用崩溃匹配或退出码匹配，容易将无关崩溃误判为目标漏洞；
  - 任务描述提供过多信息（如sanitizer报告中的函数名、行号），降低了漏洞狩猎的难度。
- **整体含义**：为了真实评估LLM在长周期、高复杂度软件漏洞狩猎中的能力，亟需一个更贴近真实场景、自演化、具有严格验证流程的基准。SEC-bench Pro正是为此而生，专注于浏览器级JavaScript引擎（V8和SpiderMonkey）中的关键漏洞。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：将公开披露的、带有具体PoC输入和修复补丁的安全报告，通过自动化三阶段流水线转化为可复现的基准任务，并利用三映像（vulnerable/fixed/latest）执行证据由LLM法官进行属性判断。
- **三阶段流水线**：
  1. **报告收集（Phase 1）**：从项目专用问题跟踪器（Chromium Issue Tracker、Mozilla Bugzilla）及额外来源（MFSA、Pwn2Own、CISA KEV）采集安全报告，提取漏洞标题、描述、bisect提交、勋章/严重性元数据、PoC工件和修复提交。过滤掉缺少PoC或修复的报告。
  2. **智能体驱动环境重建（Phase 2）**：将每个报告、PoC和结构化任务提示交给自主编码智能体（CODEX、Claude Code、Open Code等），智能体在沙箱Docker容器中驱动构建系统，生成带适当sanitizer的二进制文件，运行PoC验证崩溃签名，最终输出一组工件：Dockerfile（漏洞版和修复版）、结构化元数据、PoC、崩溃stderr、补丁目录。
  3. **自动化验证预言（Phase 3）**：对每个候选实例运行两个预言：
     - **漏洞映像预言**：在重建的漏洞容器中多次运行PoC，检查stderr是否匹配预期的错误类型（SANDBOX_VIOLATION、ASAN_CRASH、DCHECK、RUNTIME_CRASH）。
     - **修复映像预言**：应用补丁后重建二进制，多次运行同一PoC，确认所有尝试均落入“blocked”类别（不再产生目标崩溃）。只有同时通过两个预言的实例才被收入最终数据集。
- **LLM法官分级**：在评估阶段，将待测智能体生成的PoC分别在漏洞、修复、最新三个Docker映像上执行，捕获退出码、stdout、stderr。LLM法官基于三映像证据输出三个判决之一：**verified**（漏洞映像出现目标漏洞崩溃，且修复/最新映像未矛盾）、**unsure**（证据不完整，需人工裁定）、**illegal**（漏洞映像未匹配目标，或修复/最新映像出现无关崩溃）。

## 3. 实验设计：数据集、基准、对比方法

- **数据集**：SEC-bench Pro包含183个已验证漏洞实例：
  - **V8**：103个实例（其中86个有Google VRP奖金，累计超150万美元），覆盖类型混淆、释放后使用、越界读写、沙箱绕过、JIT错误、整数溢出等。
  - **SpiderMonkey**：80个实例（sec-high/sec-critical级别），覆盖类似漏洞类型，但无沙箱绕过（Spidermonkey无内进程沙箱检查）。
- **对比方法**：三种最先进的编码智能体框架：
  - **CODEX**：运行OpenAI GPT-5.4
  - **Claude Code**：运行Anthropic Opus 4.6
  - **Open Code**：运行Moonshot Kimi-K2.6（开源权重基线，仅用于V8）
- **评估设置**：每个实例给智能体90分钟预算，300秒单次执行超时，每个映像最多重试3次。主要指标为**通过法官验证的实例成功率**（至少一个PoC获得verified）。

## 6. 主要结论与发现

- **前沿智能体成功率远低于实际需求**：最佳单智能体在V8上仅为32.0%（CODEX），在SpiderMonkey上为38.8%（Claude Code）；开源基线Kimi-K2.6在V8上仅11.7%。
- **不同智能体具有互补性**：CODEX和Claude Code解决的是部分重叠的实例集；联合覆盖率在V8上升至37.9%，在SpiderMonkey上升至48.8%，说明使用集成策略可提升上限。
- **崩溃唯一匹配严重高估成功率**：使用仅检查漏洞映像崩溃的简单标尺，会将验证成功率平均膨胀43.6%（如Claude Code在SpiderMonkey上从38.8%虚增至57.5%）。法官的三映像判决有效剔除了误报。
- **失败智能体消耗更多token和时间**：成功实例往往在低/中计算预算内完成，失败实例则花费更多token（如Claude Code在V8上失败实例平均55.6M tokens，成功平均18.8M tokens），表明长期探索并非深化分析而是低效循环。
- **漏洞类型难度差异**：整数溢出/截断类最易解决；类型混淆和释放后使用虽数量多但成功率低；沙箱绕过（V8）和JIT问题（SpiderMonkey）最难。

## 7. 优点

1. **自演化流水线**：通过三阶段管道，可轻松添加新披露的PoC+补丁漏洞，无需人工重新设计基准，具备可扩展性。
2. **真实世界漏洞来源**：实例来自实际项目问题跟踪器、奖金项目、Pwn2Own和CISA KEV，涵盖已证实被利用的漏洞，具有高度现实性。
3. **三映像验证机制**：不仅利用漏洞/修复映像，还增加最新映像，并通过LLM法官进行属性判断，有效避免误报和归因错误，比传统崩溃匹配更可靠。
4. **自动化环境重建**：使用编码智能体自动重建历史构建环境，解决了手动重建负担，使大规模基准构建可行。
5. **公开数据集与Docker映像**：每个实例提供漏洞、修复、最新三个Docker映像，确保实验可复现，便于社区比较。
6. **细致的失败模式分析**：对法官判为illegal的PoC进行了分类（无崩溃、固定映像崩溃、离目标等），揭示了智能体失败的根本原因。
7. **跨引擎评估**：覆盖V8和SpiderMonkey两个竞争性JS引擎，增加了评估的普适性。

## 8. 不足与局限

1. **适用范围有限**：当前仅实例化于JS引擎，虽然框架理论上可扩展到其他项目（如Linux内核），但实际验证和评估仅在V8和SpiderMonkey上完成，通用性有待证明。
2. **智能体评估单一**：仅测试了三种智能体框架，且开源基线仅用于V8。缺少对其他前沿模型（如GPT-4o、Claude Sonnet等）以及更多样的agent架构的对比。
3. **计算资源未披露**：论文未说明具体GPU型号、数量、训练或推理时长，虽然提供了成本数据，但难以进行精确的算力横向比较。
4. **法官依赖LLM本身**：尽管有可靠性保护措施（重试、格式检查、人工裁定unsure），但LLM作为法官仍可能存在偏见或不一致，尤其在复杂边界案例中。论文虽进行了少量人工审计，但未系统评估法官的准确性。
5. **时间预算固定**：每个实例给90分钟，但未分析更短/更长预算下的表现，也没有考虑多轮反馈或人类协助的影响。
6. **缺乏零日漏洞测试**：所有实例均为已知漏洞（有修复补丁），无法评估智能体在未知漏洞上的能力。作者在Motivation部分提到CyberGym有零日发现，但SEC-bench Pro本身未设计此维度。
7. **分级粒度问题**：LLM法官只有verified/unsure/illegal三种结果，缺少部分正确的中间态（如PoC触发了目标漏洞但未完全通过三映像检查）。unsure实例依赖人工处理，可能引入主观性。
8. **基准规模仍有限**：183个实例虽然质量高，但数量相对其他数据集（如CVE-Bench、ARVO）较小，依赖于项目公开报告的可用性，未来需持续扩展。
