## 摘要
Linux内核驱动中的引用计数错误可能导致严重的资源管理不当和安全漏洞。我们介绍了DrvHorn，一种新颖的自动化工具，通过减少引用计数验证来利用Linux驱动接口进行断言检查问题。通过对Linux内核的高效建模和积极的程序切片，DrvHorn在v6.6 Linux内核的所有平台驱动中发现了545个错误，其中424个是先前未知的，其误报率（29.9%）低于先前研究。为了解决这些新发现错误的根本原因，我们向Linux内核提交了补丁，其中45个已被合并。

## Abstract
Reference counting bugs in Linux kernel drivers can lead to severe resource mismanagement and security vulnerabilities. We introduce DrvHorn, a novel automated tool to detect these bugs by reducing reference counting verification to an assertion checking problem leveraging the Linux driver interface. Through efficient modeling of the Linux kernel and aggressive program slicing, DrvHorn discovered 545 bugs, of which 424 were previously unknown, across all platform drivers in v6.6 Linux kernel, with a lower false positive rate of 29.9% compared to prior studies. To address the root causes of these newly discovered bugs, we submitted patches to the Linux kernel, and 45 of them were merged.

- **核心问题**：Linux内核驱动中的引用计数（refcount）错误会导致内存泄漏（refcount过高）或释放后使用（UAF，refcount不足），进而引发资源管理混乱和安全漏洞。例如CVE-2023-7192可被本地攻击者利用进行拒绝服务攻击。
- **研究动机**：Linux驱动代码占内核总量的约66.5%，且用户可自由加载驱动，因此自动化静态验证驱动的引用计数正确性至关重要。现有方法（如RID、LinKRID）假阳性率高（70.5%~76.6%），或需要大量手动建模（如Avinux），无法实现大规模自动化高精度检测。
- **整体含义**：本文提出DrvHorn工具，通过将引用计数验证转化为断言检查问题，利用Linux驱动接口的简单不变性实现高效、低假阳性的自动化检测，显著降低了人工审计成本。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
- **观察**：所有驱动在初始化函数（如`probe()`）中获取的资源必须在初始化失败时释放。因此可导出通用不变性：**初始化函数返回错误时，所有引用计数的净变化必须为零**。
- **方法**：将驱动初始化函数的引用计数验证归约为**断言检查问题**：生成一个`main()`函数，调用目标驱动初始化函数，并在错误返回路径上断言设备、设备树节点、固件节点等所有引用计数的增量均为0。

### 关键技术细节
1. **归约为断言问题**：生成LLVM bitcode格式的检测代码，包含对`∆(refcount of devices)==0`等断言。
2. **Linux内核建模**：
   - **设备树API建模**：将设备树API分为四类（返回NULL/非NULL，递减/不递减引用计数），每类统一建模，降低复杂度。
   - **设备与总线建模**：对`bus_find_device()`等返回引用计数增长的结构体（如`struct usb_interface`）进行建模，避免指针别名导致假阳性。
   - **设备资源管理（Devres）建模**：简化`__devm_add_action()`为立即执行清理函数，不显式跟踪devres列表。
   - **解决内联汇编**：将105种x86-64内联汇编指令替换为等价LLVM IR；不支持的替换为非确定值。
3. **激进程序切片**：
   - **第一步**：标记哪些函数参数是“essential”（嵌入`struct kref`且函数体可能修改引用计数）。
   - **第二步**：从每个函数的`call`和`ret`指令开始，递归收集必要指令（条件包括：控制流指令、参数底层对象为essential的调用、加载/存储依赖、所有操作数必要的指令）。其余指令替换为非确定值。
   - 该切片比LLVM的DCE更激进，保留引用计数相关操作，移除无关代码（如`inode_lock`等），使验证规模可控。
4. **后端验证**：采用SeaHorn（基于约束霍恩子句CHC的验证器），因其擅长处理循环（驱动中常见）。对比了SeaBMC（有界模型检查器）作为替代方案。
