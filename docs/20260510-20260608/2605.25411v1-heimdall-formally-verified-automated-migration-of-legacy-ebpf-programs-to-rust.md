## 摘要
扩展伯克利数据包过滤器（eBPF）程序是用于Linux内核中网络、可观测性和安全执行的核扩展。内核中的eBPF验证器检查eBPF程序的底层内存安全性和终止性，但它并不强制许多更高级别的源级属性，例如初始化规范、模式一致性或错误处理。我们记录了六类源级错误，这些错误可以编译通过、通过内核验证器，并可能静默地破坏数据、将先前跟踪的事件泄露到用户空间，或产生错误的执行结果。其中，我们在十个开源eBPF程序中发现了先前未报告的信息泄露，这些程序的环形缓冲区或栈驻留事件记录携带可完全解码的先前跟踪事件，包括用户标识路径和重复的内核文本返回地址，足以在每个事件中恢复KASLR偏移量，进入用户空间。为了加固这些已被验证器接受的存在错误的程序并支持安全迁移，我们提出Heimdall，一个自动化流水线，使用大型语言模型将遗留的libbpf C程序翻译为Aya Rust。Heimdall迭代地修复编译和内核验证器失败，通过静态分析安全引擎拒绝Rust-Aya中的不安全逃逸机制，并通过符号执行和基于Z3的等价性检查证明每个程序与原始程序的等价性。在102个eBPF程序中，Heimdall生成了96个形式上证明等价的翻译（94.1%）。Heimdall是第一个自动化将生产级eBPF程序迁移到内存安全语言的系统，并为每个程序提供形式化保证，确保迁移保留了可观察行为。

## Abstract
Extended Berkeley Packet Filter (eBPF) programs are kernel extensions used for networking, observability, and security enforcement in the Linux kernel. The in-kernel eBPF verifier checks low-level memory safety and termination on eBPF programs, but it does not enforce many higher-level source-level properties, such as initialization discipline, schema consistency, or error handling. We document six classes of source-level bugs that compile, pass the kernel verifier, and can silently corrupt data, leak previously traced events to userspace, or yield incorrect enforcement outcomes. Among these, we identify previously unreported information leaks in ten open-source eBPF programs whose ring-buffer or stack-resident event records carry fully decodable prior traced events, including user-identifying paths and recurring kernel-text return addresses sufficient to recover the KASLR slide on every event, into userspace. To harden such verifier-accepted buggy programs and support safe migration, we present Heimdall, an automated pipeline that uses large language models to translate legacy libbpf C programs to Aya Rust. Heimdall iteratively repairs compilation and kernel-verifier failures, rejects unsafe escape hatches in Rust-Aya with a static analysis safety engine, and proves per-program equivalence to the original via symbolic execution and Z3-based equivalence checking. Across 102 eBPF programs, Heimdall produces 96 formally proven-equivalent translations (94.1%). Heimdall is the first system to automate memory-safe-language migration of production eBPF programs with per-program formal guarantees that the migration preserves observable behavior.

## 1. 核心问题与整体含义（研究动机和背景）

Linux 内核中的 eBPF（扩展伯克利数据包过滤器）程序广泛用于网络、可观测性、安全策略执行等场景。内核自带的 eBPF 验证器确保程序在低层级的内存安全性和有界执行，但**不强制许多更高级别的源级属性**，例如：

- 变量/结构体初始化规范
- 错误处理（如助手函数返回值是否被检查）
- 输出模式一致性
- 地图类型/模式一致性
- 符号/无符号类型混淆

论文**记录了六类这样的源级错误**，这些错误能顺利通过编译和内核验证器，但可导致严重问题：
- 静默数据损坏
- 将先前跟踪的事件信息泄露到用户空间
- 输出错误的执行结果

尤其严重的是，论文在十个开源 eBPF 程序中发现了**此前未报道的信息泄露漏洞**：环形缓冲区或栈上驻留的事件记录携带了可完全解码的先前跟踪事件，包括用户标识路径和可重复的内核文本返回地址，足以在每个事件中恢复 KASLR（内核地址空间布局随机化）偏移量，从而彻底绕过 KASLR 保护。

**整体含义**：依赖内核验证器作为唯一安全屏障存在重大缺陷。将 eBPF 程序迁移到内存安全语言（如 Rust）可以从源级别关闭这些安全漏洞，但手工重写庞大的 libbpf C 生态系统不现实。因此，需要自动化迁移工具，并必须提供**形式化保证**，确保迁移后的程序与原始程序行为等价。

## 2. 方法论：核心思想、关键技术细节、算法流程

### 核心思想
Heimdall 是一个**自动化流水线**，用 LLM 将遗留的 libbpf C 程序翻译为 Aya Rust（一个纯 Rust 实现的 eBPF 库），并通过以下四种机制确保安全性和等价性：
1. **编译和内核验证器修复**：迭代修复 LLM 生成的 Rust 代码中的编译错误和内核验证器拒绝。
2. **安全静态分析**：拒绝不安全和不习惯的 Rust-Aya 模式（如未类型化的环形缓冲区保留、未检查的助手返回值等）。
3. **符号执行**：对 C 和 Rust 编译后的 eBPF 字节码进行符号执行，生成结构化公式（路径谓词、返回值、地图状态）。
4. **等价性检查**：使用 Z3 SMT 求解器检查两个程序在所有输入上是否语义等价。

### 关键技术细节

- **LLM 翻译 + 迭代修复**：输入 C 源文件、入口点符号名、地图名与类型。LLM 根据含 Aya API 映射、钩子宏表、示例的提示生成 Rust。迭代修复编译错误和内核验证器错误。
- **安全静态分析**：基于模式的源码级安全引擎，剔除 11 类不安全模式（如 `mem::transmute`、未处理的 `Result`、未填充的环形缓冲区、非原子的原始指针写入等）。
- **eBPF 符号执行后端（基于 angr）**：论文构建了完整的 eBPF 后端，覆盖 56 个内核助手模型、20 种地图类型、BPF-to-BPF 调用、原子操作等。
- **条件等价性定义**：由于安全改进的 Rust 翻译（如零初始化、错误处理）在触发原始 C 错误的那部分输入上行为不同，论文引入**条件等价**概念：只在满足安全条件的输入上要求 C 和 Rust 程序等价。安全条件包括两条：
  - 所有助手调用成功（`Φ_safe^(i)`）
  - 输出 sink 内容不参与比较（`Φ_safe^(ii)`），即允许 Rust 翻译零填充未初始化内存等改进。
- **ITE 链地图编码**：对于基于哈希的地图，用 If-Then-Else 链表示每个路径上的写入序列，并跨路径组合，从而精确建模 last-write-wins 语义。
- **等价性检查**：Z3 求解否定后的等价公式，若 UNSAT 则等价；若 SAT 则生成反例，反馈给 LLM 修复。

### 算法流程（5 阶段，图 1）
```
Stage 1: LLM 翻译 C → Rust Aya
Stage 2: 编译 + 内核验证（迭代修复）
Stage 3: 安全静态分析（拒绝不安全模式）
Stage 4: 符号执行（angr 生成公式）
Stage 5: Z3 等价性检查（若 SAT 则返回反例给 Stage 1 修复）
```
同时支持**确定性流水线**（脚本驱动，LLM 只翻译）和**智能体流水线**（Agentic，LLM 自由使用工具和子代理）。

## 6. 主要结论与发现

1. **验证器存在不可忽视的空缺**：6 类源级错误可绕过验证器，其中未初始化状态导致的 KASLR 泄露和信息泄露在 10 个生产级 eBPF 程序中被证实（包括 bashreadline、mountsnoop 等）。
2. **编译成功不是翻译成功的代理指标**：所有 Baseline 方法都达 100% 编译，但全通过率仅 4~12%。
3. **内核验证通过也不是安全或等价性的充分条件**：Baseline 的 KV 通过率 71~96%，但安全通过率降至 41~53%，等价通过率降至 12~22%。
4. **Heimdall 显著提升翻译成功率**：Heimdall (Agentic) with opus-4.6 在 51 个程序上达到 98% 全通过率，在 102 个程序上达 94.1% 全通过率。
5. **Heimdall 能有效关闭发现的漏洞**：对 96 个已验证翻译的 C 源扫描显示，所有观察到的未初始化状态、未检查助手返回、符号/无符号混淆实例均被 Rust 翻译关闭（100%）。
6. **运行时开销可接受**：在 10 个程序的 1,000 次配对试验中，所有事件记录匹配，运行时开销中位数为 1.33×（范围 0.44×~3.60×）。
7. **智能体方法优于确定性方法**：agentic 变体达到更高成功率（98% vs 63%），但代价是更高的 token 消耗（~2.6M vs ~286K 输入 token）和成本。

## 7. 优点

- **首个自动化迁移 eBPF 程序并提供形式化保证的系统**：填补了 C 语言 eBPF 到 Rust 迁移的空白，且不是仅靠模糊测试，而是基于 SMT 求解的字节码级等价性证明。
- **条件等价性**的引入合理解决了安全改进与等价性的冲突：允许 Rust 翻译在原始 C 有 bug 的输入上行为更好，同时保证其他输入等价。
- **完整的端到端自动化**：从 C 输入到等价验证，所有步骤（翻译、修复、安全分析、符号执行、等价检查）集成在单一循环中，无需人工干预。
- **符号执行后端贡献**：构建了覆盖完整 eBPF 指令集、56 个助手、20 种地图类型、原子操作等的 angr 后端，可独立用于其他 eBPF 分析任务。
- **详尽的漏洞报告**：发现并验证了 10 个程序的未初始化泄露，包括 KASLR 绕过，并提供了可重现的复现步骤。
- **静态安全分析引擎**：针对 eBPF 特有模式（如未类型化环缓冲、非原子指针写入、`get_stackid` 错误处理）制定规则，比通用 Rust 编译器检查更精准。
- **实验充分且透明**：评估覆盖 102 个程序，公开所有翻译、脚本和验证产物，便于复现和扩展。

## 8. 不足与局限

1. **扩展性限制**：符号执行路径探索上限为 50,000，对复杂程序（深嵌套分支、大量地图写入）会超时。6 个程序无法完全验证（3 个部分验证+2 个超出求解能力）。依赖 Z3 求解大规模 ITE 链公式在某些情况下不可扩展。
2. **安全策略是手写规则**：基于模式匹配，只能捕捉已预定义的模式，不能防范未预见的 idiomatic 问题或新发现的安全漏洞。可能漏检某些巧妙的不安全 Rust 写法。
3. **助手建模有限**：仅建模 56 个内核助手，而内核暴露 100+ 个。若翻译调用了未建模的助手，等价性检查不覆盖该部分行为。
4. **条件等价性引入的灰色区域**：虽然合理，但默认不比较输出 sink 内容，意味着虽然 Rust 翻译消除了泄露，但等价性检查未正式证明这一点。用户需依赖单独的安全分析报告。
5. **原子性检测为粗粒度启发式**：静态扫描比较原子指令总数，而非逐点证明原子性保留，可能产生误报或漏报。
6. **运行时验证范围有限**：仅测试 10 个程序，且比较的是过滤后记录数而非完整字节流（因时间戳、PID 等字段总是变化）。主要验证了等价性（记录数匹配），但运行时开销测量可能受硬件平台影响。
7. **应用限制**：Aya 不支持 `cgroup_array` 地图、USDT 参数、遗留 `BPF_LD_ABS/IND` 指令，因此 17 个程序（占 14.3%）无法翻译。对于依赖这些特性的程序，Heimdall 不适用。
8. **成本较高**：智能体翻译每个程序平均耗时 23.3 分钟、$4.05（基准测试子集）；全量评估平均 $2.94/程序。对于大型代码库，总成本可能显著。但相比手工重写，仍可接受。
9. **无真实世界部署验证**：未在长期运行的 production 环境中测试翻译后的 Rust Aya 程序的稳定性、性能稳定性或安全改进的实际效果。
