## 摘要
Linux内核驱动中的引用计数错误可能导致严重的资源管理不善和安全漏洞。我们提出DrvHorn，一种新颖的自动化工具，通过将引用计数验证简化为利用Linux驱动接口的断言检查问题来检测这些错误。通过对Linux内核的高效建模和激进程序切片，DrvHorn在v6.6 Linux内核的所有平台驱动中发现了545个错误，其中424个是先前未知的，误报率（29.9%）低于先前研究。为了解决这些新发现错误的根本原因，我们向Linux内核提交了补丁，其中45个已被合并。

## Abstract
Reference counting bugs in Linux kernel drivers can lead to severe resource mismanagement and security vulnerabilities. We introduce DrvHorn, a novel automated tool to detect these bugs by reducing reference counting verification to an assertion checking problem leveraging the Linux driver interface. Through efficient modeling of the Linux kernel and aggressive program slicing, DrvHorn discovered 545 bugs, of which 424 were previously unknown, across all platform drivers in v6.6 Linux kernel, with a lower false positive rate of 29.9% compared to prior studies. To address the root causes of these newly discovered bugs, we submitted patches to the Linux kernel, and 45 of them were merged.

## 1. 核心问题与整体含义（研究动机和背景）
- **核心问题**：Linux 内核驱动中的引用计数（refcount）错误会导致严重的安全漏洞，如内存泄漏（过度引用）或释放后使用（UAF，引用不足）。CVE-2023-7192 等实例表明这类漏洞可被本地攻击者利用进行拒绝服务攻击。
- **背景**：Linux 内核约 66.5% 的代码是驱动程序，且用户可自由加载自定义驱动。参考计数被广泛用于手动管理内核对象的生命周期，但人工维护极易出错。现有的静态检测方法（如 RID、LinKRID）误报率高（43.5%~76.6%），或需要大量人工介入（如 Avinux），缺乏可扩展且精准的自动化方案。
- **研究动机**：开发一种自动、可扩展且低误报的引用计数错误检测工具，专注于 Linux 驱动，利用驱动接口的规律性简化验证问题。

## 2. 方法论：核心思想、关键技术细节
- **核心思想**：将引用计数验证问题转化为一个断言检查问题，该断言对所有 Linux 驱动通用：**在驱动初始化函数失败时，所有引用计数的净变化应为零**（即所有获取的引用必须被释放）。
- **关键技术细节**：
  1. **归约为断言问题（Section 3.1）**：生成一个主函数，调用驱动的初始化函数，若返回错误则对所有关注的对象（设备、设备树节点、固件节点等）断言其引用计数差值等于 0。
  2. **内核建模（Section 3.2）**：为关键内核 API 提供精简但保真的抽象，例如：
     - 设备树 API：按行为模式分为四类（返回 NULL / 非 NULL、递减引用计数 / 不递减），统一建模，避免复杂指针分析。
     - 设备和总线 API：建模 `bus_find_device()` 等返回引用递增的设备指针的函数，确保返回的内存区域不别名化。
     - 资源管理（Devres）：将 `__devm_add_action()` 建模为立即插入清理函数调用，避免追踪复杂的 devres 链表。
  3. **激进程序切片（Section 3.3）**：在 LLVM IR 级别执行，仅保留与引用计数相关的指令（例如函数参数中嵌入 `struct kref` 且函数体可能修改计数的调用，以及必要的数据依赖指令）。非必要指令被替换为非确定性值，同时保留 `writeonly` 属性以避免丢失副作用。相比普通死代码消除（DCE）更激进，使验证可行。
  4. **求解器选择**：采用 SeaHorn（基于 CHC 的验证器），因其能更好地处理循环（驱动中常见），并支持递归和复杂数据流。内联策略用于多数调用，剩余递归调用由 CHC 求解。
- **流程概览**：  
  1. 将 Linux 内核源码编译为 LLVM 位码。  
  2. 根据目标驱动名称进行内核建模（替换 API 实现）。  
  3. 执行激进切片。  
  4. 插入断言代码，形成断言检查问题。  
  5. 交给 SeaHorn 求解，输出是否存在可能的引用计数错误。

## 6. 主要结论与发现
- **有效性**：DrvHorn 在 3387 个平台驱动中报告了 777 个 bug，545 个为真实（假阳性率 29.9%），其中 424 个是之前未知的。
- **实际影响**：45 个提交的补丁已被 Linux 主线内核合并。
- **低漏报率**：在 v6.6 发布后修复的 17 个适用已知 bug 中，DrvHorn 检测到 14 个（2 个超时，1 个因切片漏报），漏报率较低。
- **优于现有方法**：假阳性率远低于 RID（76.6%）、CID（70.5%）、LinKRID（43.5%）。相比 SeaBMC（漏检 39.4%，超时 24.2%），CHC 方法更具优势。
- **切片必要性**：普通 DCE 会导致所有 545 个真实 bug 的驱动超时，而激进切片使验证可行。

## 7. 优点
- **方法创新**：将引用计数验证归约为一个简洁的断言，有效利用驱动接口的规律性，避免复杂的内核全局分析。
- **实用性强**：不仅发现大量未知 bug，还积极提交合并补丁，部分错误通过引入新的通用 API（如 `devm_device_init_wakeup()`）从根本上解决多驱动共同问题。
- **设计可扩展**：工具可与其他 LLVM IR 求解器集成；建模方法模块化（设备树 API 分类），易适应新 API。
- **低假阳性**：通过针对性建模和切片，假阳性率（29.9%）比先前工作显著降低，利于实际部署。

## 8. 不足与局限
- **覆盖范围有限**：仅验证初始化失败路径，未覆盖成功路径、运行阶段、清理函数的所有错误模式（尽管论文论证了初始化错误路径已能捕获多数情况，但未完全证明）。
- **依赖切片精确性**：27 个假阳性源于切片导致的间接信息流丢失；1 个漏报也由此造成。切片策略并非完全保真。
- **指针分析保守**：205 个假阳性源于 SeaHorn 的内存模型对可能别名的过度近似。
- **可扩展性限制**：807 个驱动超时（占比 23.8%），尤其是包含复杂嵌套循环的函数（如 `__mtk_clk_simple_probe`），CHC 求解器难以处理。
- **仅支持 x86-64**：内联汇编建模仅处理了 105 种 x86-64 指令，其他架构未验证。
- **不适用于非 platform 驱动**：论文仅测试了平台驱动，声称可扩展但未定量验证其他类型驱动的效果。
- **未提供算力成本**：无总运行时间、能耗等，不利于资源规划。
