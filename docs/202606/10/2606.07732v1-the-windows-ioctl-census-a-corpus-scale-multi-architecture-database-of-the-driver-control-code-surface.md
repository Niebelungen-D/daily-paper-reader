## 摘要
Windows驱动程序通过I/O控制（IOCTL）码暴露其内核，而缓冲区后一个未检查的长度就能将非特权调用转化为内核写入。研究社区对此接口拥有强大的扫描器以及一份已知恶意驱动程序的精选列表，但缺乏该接口本身的地图。我们构建了这张地图。Windows IOCTL普查是一个可查询的数据库，包含27,087个已签名Windows驱动程序的控制码分发接口，通过一次确定性的、架构无关的传递（无需符号执行）恢复。通过读取提升后的中间表示而非运行符号引擎，它能在x86和x64架构下恢复整个语料库80%的分发接口，包括现有扫描器放弃处理的32位部分。在64位路径上，它还增加了处理程序可达性、污点分析以及调用图。一个大型语言模型（LLM）对可达处理程序进行优先级排序以辅助分类。我们将此普查作为包含数千万行数据的公共数据集发布：27,087个二进制文件、310万个解码控制码、818万个函数以及1595万个调用边。

## Abstract
A Windows driver exposes its kernel through I/O control (IOCTL) codes, and a single unchecked length on the buffer behind one turns an unprivileged call into a kernel write. The research community has strong scanners for this surface and a curated list of known-bad drivers, but no map of the surface itself. We build that map. The Windows IOCTL Census is a queryable database of the control-code dispatch surface of 27,087 signed Windows drivers, recovered by one deterministic, architecture-neutral pass with no symbolic execution. Reading a lifted intermediate representation instead of running a symbolic engine lets it recover a dispatch surface for 80% of the corpus across x86 and x64, including the 32-bit half existing scanners abort on. On the 64-bit lane it adds handler reachability, taint, and the call graph. An LLM ranks the reachable handlers for triage. We release the census as a public dataset of tens of millions of rows: 27,087 binaries, 3.1M decoded control codes, 8.18M functions, and 15.95M call edges.

# Windows IOCTL Census 论文详细总结

## 1. 论文的核心问题与整体含义（研究动机和背景）
Windows 驱动程序通过 I/O 控制码（IOCTL）暴露内核接口，用户态程序可通过 `DeviceIoControl` 调用这些接口。如果某个未检查长度的缓冲区被非特权调用者利用，就可能造成内核读写漏洞（如 BYOVD 攻击）。当前研究存在两个孤立分支：
- **扫描器**（如 ScrewedDrivers、POPKORN、ioctlance）：使用符号执行等技术，以单次扫描输出漏洞列表，但丢弃了恢复过程中的结构信息，且仅支持 x64（32 位驱动完全无法处理）。
- **精选列表**（如 LOLDrivers、微软驱动黑名单）：手工维护已知漏洞驱动，但覆盖范围极其有限，无法反映整个驱动生态的长尾。

**缺失的中间层**：一个持久、可查询的驱动程序控制码接口数据库，同时支持静态分析、跨驱动查询以及后续漏洞研究。本文构建了 **Windows IOCTL Census**——涵盖 27,087 个已签名 Windows 驱动程序的全面控制码分发接口地图。

## 2. 论文提出的方法论：核心思想、关键技术细节
**核心思想**：使用**确定性、架构无关的二进制提升（lifting）**，通过一次遍历恢复驱动 IOCTL 调度表面，完全不依赖符号执行，从而避免路径爆炸问题。整体流程分为三个阶段（Recover → Enrich → Rank）：

### Recover 阶段（确定性）
- **调度与码恢复**：识别 `IRP_MJ_DEVICE_CONTROL` 分发例程，通过 `cmp-immediate` 或跳转表模式提取控制码常量，并解码为（设备类型、传输方法、所需访问、功能号）。使用 **glaurung** 的架构中立中间表示（IR），同一识别器在 x86 和 x64 下均生效。
- **可达性**：构建整个驱动的过程间调用图，从分发例程出发计算前向闭包，记录每个函数是否可达以及距离。
- **污点分析（仅 64 位）**：轻量级污点传播，种子点为 `Irp->AssociatedIrp.SystemBuffer`（缓冲输入指针），标记所有到达该值的解引用或存储位置，并记录是否存在比较净化。该分析为过程内、固定窗口。
- **元数据**：提取 PE 版本信息、嵌入的设备安全描述符（SDDL）、导入导出等。

### Enrich 阶段（模型辅助）
- 仅对满足以下条件的处理程序运行 LLM（GPT-5.5/Claude Opus 4.7）：位于**宽松设备描述符**下且携带**未净化的缓冲输入接收器**。
- LLM 接收特征摘要（非原始字节）：ACL、解码码、污点接收器、结构错误类命中、反汇编片段。输出风险等级、原语类型、CWE 集等结构化裁决。

### Rank 阶段
- 通过关系视图（如 `v_unaudited_unpriv_surface`）对未审计的接口进行排序，支持跨驱动、跨构建的查询。

## 6. 论文的主要结论与发现
- **成功构建大规模 IOCTL 数据库**：覆盖 27087 个驱动，恢复 80%（21708 个）的调度表面，其中 x86 驱动贡献了 10927 个（此前符号工具无法处理）。
- **确定性恢复优于符号执行**：在交叉对比中，本文静态污点标记了 231 个驱动存在未净化接收器，而 ioctlance 仅标记 54 个。两者在 ioctlance 完成驱动的重叠率为 65%。分歧主要由符号执行无法进入处理程序（171 个驱动 ioctlance 未发现任何处理程序）导致。
- **LLM 辅助分类降低人力成本**：从 330 个高级候选处理程序（由确定性过滤得到）中，LLM 评定 135 个为“严重”、159 个为“高”，但大多数（78/135）对应已有公开 CVE 或已知审计的表面，并非全新漏洞。
- **架构中立的优势**：同一识别器在 x86 和 x64 均工作，x86 的恢复经反汇编确认（272/272 代码正确），但 x86 仅做表面恢复，无污点分析。
- **发布数据**：开放结构层数据集（含 310 万编码码、818 万函数、1595 万调用边），保留目标层（接收器、LLM 裁决等）。

## 7. 优点
- **确定性、架构无关、无路径爆炸**：通过静态 IR 提升而非符号执行，同时处理 x86 和 x64，覆盖 32 位遗留驱动（符号工具无法做到）。
- **持久化结构而非一次性列表**：存储调用图、可达性、解码码等，支持跨驱动查询（例如相同解码码的驱动集合），比扫描器输出更灵活。
- **低成本 LLM 辅助**：仅在确定性过滤后的少量候选上运行模型，避免对全量数据做昂贵推断，且模型输入是特征摘要而非原始字节，提高忠实度。
- **开放源码与数据**：glaurung 引擎开源；结构层数据集公开，鼓励社区使用。
- **交叉验证能力**：引入 ioctlance 结果并与静态污点联合，暴露各方法盲区（如静态污点无法检测 METHOD_NEITHER 和 double-free）。

## 8. 不足与局限
- **深度分析仅限 x64**：x86 驱动仅恢复调度表面和编码码，无污点/调用图分析；Arm64 仍为元数据级。x86 污点源已实现但未经验证和持久化。
- **污点源覆盖不全**：仅针对 WDM 模型的缓冲输入（SystemBuffer），缺少 METHOD_NEITHER、直接 I/O 的 MDL 缓冲区、框架驱动（KMDF/WDF）的 `WdfRequestRetrieveInputBuffer` 路径。因此许多已知漏洞（物理内存映射、寄存器读写）无法被检出。
- **设备 ACL 静态读取有误报/漏报**：嵌入 SDDL 可能绑定其他对象，运行时 ACL 可能由 INF 设置，导致部分可达驱动被错误归为不可达（如 NDKPing）。
- **模型层级无真阳性率**：135 个“严重”判决多为重复标记已有 CVE 的表面，并非新发现漏洞实际验证。
- **回测样本极小**：已知漏洞驱动大多不在语料库中（语料库以微软更新目录为主），无法有效评估系统对现代 BYOVD 驱动的探测能力。
- **符号解析未完成**：大部分函数名未解析（依赖外部数据库未完成）。
- **排名未区分新漏洞与已知审计**：最高排名中包含大量已审计的驱动（如 NVIDIA、Realtek 等），需要额外的披露过滤器。
