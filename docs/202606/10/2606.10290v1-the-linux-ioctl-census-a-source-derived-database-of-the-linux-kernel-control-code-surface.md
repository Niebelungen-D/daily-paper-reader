## 摘要
ioctl 系统调用是 Linux 通用的设备控制接口。用户态程序打开设备节点，向驱动程序传递一个数值命令码和一个参数缓冲区，驱动程序根据该码执行相应操作，无论是配置硬件、读取状态，还是在内核内外移动数据。驱动程序自行定义这些命令，数量达数千，并在内核上下文中解析其参数，这使得 ioctl 处理程序成为内核中最广泛且最不统一的局部攻击面之一。一个信任未经校验的参数长度的处理程序可能越界读写内核内存，而命令空间没有集中编目。我们提出了 Linux IOCTL 普查，这是一个基于源码且可查询的攻击面清单。allmodconfig 构建跨 169 个子树编译了 878 个模块，在其上，对内核源码执行一次确定的 libclang 遍历，恢复了 586 个 ioctl 调度入口点、1,289 个解码的 _IOC 命令码、3,583 个受控输入接收点以及 1,298 个权限门。第二次遍历将内核自身文档化的威胁模型编码为一个可查询的列，将未受能力门控的 ioctl 表面（非特权可达性的上界，而非经证实的可达性）与硬能力门排除的部分分开。我们针对近期 22 个树内 ioctl CVE 对普查进行了回测，并将结构层级以开放数据形式发布，采用与配套的 Windows IOCTL 普查共享的模式，使得单个查询可跨两个操作系统。

## Abstract
The ioctl system call is Linux's catch-all device-control interface. A userspace program opens a device node and hands the driver a numeric command code and an argument buffer, and the driver does whatever that code means, whether configuring hardware, reading back state, or moving data into and out of the kernel. Drivers define these commands themselves, by the thousand, and parse their arguments in kernel context, which makes ioctl handlers one of the broadest and least uniform local attack surfaces in the kernel. A handler that trusts an argument length it never validates can read or write kernel memory out of bounds, and the command space is catalogued in no central place. We present the Linux IOCTL Census, a source-derived and queryable inventory of that surface. An allmodconfig build compiles 878 modules across 169 subtrees, and over them a single deterministic libclang pass over the kernel source recovers 586 ioctl dispatch entry points, 1,289 decoded _IOC command codes, 3,583 controlled-input sinks, and 1,298 permission gates. A second pass encodes the kernel's own documented threat model as a queryable column, separating the capability-ungated ioctl surface, an upper bound on unprivileged reach rather than proven reach, from the part a hard capability gate puts out of scope. We backtest the census against 22 recent in-tree ioctl CVEs and release the structural tier as open data, on a schema shared with the companion Windows IOCTL Census so a single query spans both operating systems.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：Linux 内核中的 `ioctl` 系统调用是设备控制的主要接口，但驱动程序各自定义成千上万的命令码，并在内核上下文中解析用户参数。由于缺乏集中编目，且大量处理程序可能未正确校验参数长度，导致越界读写等安全漏洞频发。攻击面大且不统一，缺乏系统性的枚举与威胁建模。
- **研究动机**：现有方法（如 syzkaller 动态模糊测试）能发现具体漏洞，但无法描述未触及的命令空间；人工维护的威胁模型文档（如 `Documentation/process/threat-model.rst`）是文字描述而非可查询的数据结构。因此需要一种基于源码的、可持久化查询的 ioctl 攻击面清单，以填补静态枚举与动态测试之间的空白。
- **整体含义**：该论文提供了一个名为 **Linux IOCTL Census** 的开放数据库，能够系统性地枚举所有编译模块中的 ioctl 调度入口、命令码、受控输入接收点及权限门，并将内核文档化的威胁模型编码为可查询列，从而支持跨操作系统的安全分析。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：通过 **Extract-Gate-Rank** 三阶段流水线，基于 libclang 对内核源码进行确定性静态分析，提取 ioctl 相关结构，并融合内核威胁模型给出可达性上界。
- **关键技术细节**：
  - **Extract 阶段**：利用 libclang 从编译数据库（`compile_commands.json`）构建知识图谱，识别绑定到 `file_operations.unlocked_ioctl`、`compat_ioctl`、`proc_ops.proc_ioctl` 的调度函数。然后解析 switch-case 中的 `_IOC` 宏，将命令码解码为方向、类型、序数、参数大小四个字段（图 2）。对于每个处理程序，记录其受控输入接收点（如 `copy_from_user` 等 8 种原语）及路径上的权限门（`capable`/`ns_capable`、LSM 钩子等）。接收点是否“未消毒”定义为同一函数中该行前面没有硬边界检查，这是一个文本顺序启发式，非控制流证明。
  - **Gate 阶段**：将内核威胁模型文档化的边界编码为“能力-无门控”（`capability_ungated`）列和“在威胁模型中”（`in_threat_model`）列。排除带有硬 `CAP_SYS_ADMIN` 门控、staging 驱动、debugfs 等路径。这是一个必要条件上界过滤器，而非可达性分类器。
  - **Rank 阶段**：提供若干关系视图，如按未消毒接收点数量排序的无门控调度器、覆盖率统计、跨模块共享命令码视图。
- **数据存储**：所有结果以 `(module_sha256, symbol)` 为主键的关系数据库存储，兼容 Windows IOCTL Census 模式，使得单一查询可跨两个操作系统。

## 6. 论文的主要结论与发现

- **主要发现**：
  - 在 878 个编译模块中，识别出 **586 个 ioctl 调度器**、**1,289 个解码命令码**（解析率 80%，未解析的为传统数字常量）、**3,583 个受控输入接收点**（其中 3,201 个被代理标记为“未消毒”）、**1,298 个权限门**。
  - 威胁模型过滤器：337 个 ioctl 模块中，**50 个被硬能力门控排除**，**281 个被标记为能力无门控**（非特权可达性上界），6 个为 staging 驱动。
  - 权限门集中在 `CAP_SYS_ADMIN`（512 次），这是内核威胁模型排除的关键能力。
  - 跨模块共享命令码分析显示：watchdog 的 `WDIOC_*` 命令在 31 个驱动中重复，且有魔术字节冲突。
  - Arm64 重建仅增加 1 个模块，无新命令码，表明 x86_64 覆盖了 99.7% 的 ioctl 模块。
  - CVE 回测：22 个 CVE 全部位于覆盖子树内，7 个处理程序被恢复（6 个命令码被解码），15 个丢失均因注册模型限制（DRM/V4L2 ops-table、tty 线路规程、子处理函数）。过滤器正确接纳 5 个 CVE，2 个节点-DAC 误接纳（符合预期）。
- **核心结论**：该普查是首个基于源码、可查询的 Linux ioctl 攻击面数据库，填补了静态枚举与动态测试之间的空白，并提供了跨操作系统的统一模式。

- **系统性**：对全内核（allmodconfig）进行确定性单次扫描，覆盖广，可重现。

- **开放性**：公开结构层级数据（模块、调度器、命令码、门控），保留攻击性的接收点等以避免双用途风险，体现了负责任的数据发布。

## 8. 不足与局限

- **实验覆盖**：
  - 仅针对 `x86_64 allmodconfig` 和 `arm64`，未覆盖所有架构（如 mips、riscv、s390）可能存在的专有驱动。
  - 构建覆盖率 84%，剩余 16% 主要为其他架构平台代码及所有不兼容 `allmodconfig` 的驱动。
  - CVE 回测仅 22 个，且全部在覆盖子树内；对于未覆盖子树中的漏洞无法检验。
- **方法局限**：
  - **注册模型限制**：无法处理 ops-table 调度（DRM、V4L2、ALSA）、tty 线路规程、低于调度器的子处理函数。这些正是所有 15 个 CVE 丢失的原因，是最大的单一局限。
  - **可达性上界**：能力无门控仅考虑 handler 内部的门控，未考虑节点 DAC（设备节点权限）或运行时可打开性，因此 281 个模块是总上界，可能包含实际不可达的条目。
  - **消毒代理**：`未消毒`标记仅依据同一函数内文本顺序之前的硬边界检查，未进行控制流分析或跨函数验证，精度未知，不能作为漏洞计数。
  - **静态分析固有限制**：无法发现运行时条件（竞态、状态依赖）导致的漏洞。
- **偏差风险**：
  - 命令码解析率 80%，未解析的 20% 为传统数字常量，可能包含重要命令。
  - `CAP_SYS_ADMIN` 门控占主导，但论文未分析这些门控是否真实保护所有路径（例如，是否存在绕过）。
- **应用限制**：
  - 作为数据集而非工具，不提供自动化漏洞检测；需要人工分析或与其他工具结合。
  - 可选 LLM 富化层表现一般（7 个正控制仅 4 个触发），不适合用于高精度自动化筛选。
