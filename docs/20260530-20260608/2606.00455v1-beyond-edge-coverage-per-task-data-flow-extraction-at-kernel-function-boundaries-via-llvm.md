## 摘要
基于覆盖的内核模糊测试工具（如syzkaller）依赖边覆盖（trace-pc）作为唯一的反馈信号。这种上下文无关的方法无法区分仅在参数值上有所不同的执行路径。例如，两次调用copy_from_user()时，不同的size参数会命中相同的基块，却具有截然不同的安全影响。我们提出了BOUNDARY FLOW，一个基于LLVM的插桩框架，通过提取函数参数和返回值的数流来扩展Linux KCOV。一个编译器pass（-fsanitize-coverage=dataflow-args, dataflow-ret）在函数入口处发射轻量级回调，捕获结构化的元组<PC, arg_idx, arg_size, ptr, offsets[]>，在返回处捕获<PC, ret_size, ptr, offsets[]>。复合类型通过DWARF DICompositeType元数据自动分解，无需任何源代码注释。一个独立的内核设备（/sys/kernel/debug/kcov_dataflow）提供无锁的每任务环形缓冲区，不影响现有的KCOV或syzkaller基础设施。我们展示了双重用途：模糊测试器获得状态感知的反馈，用于引导突变进入值依赖的状态转换；安全分析师获得确定性的参数记录，用于根因分析，而无需printk或kprobe的开销。一个后编译流水线（rustc, opt, llc）使得Rust内核模块插桩成为可能，无需修改rustc，这是在drgn/vmcore因-O2 DWARF省略而失败的情况下唯一运行时捕获Rust函数参数的方法。我们在五类漏洞（OOB、UAF、double-free、10深链传播、Rust FFI、Rust for Linux Modules）上进行了评估，插桩路径的开销小于3%。

## Abstract
Coverage-guided kernel fuzzers such as syzkaller rely on edge coverage (trace-pc) as their sole feedback signal. This context-blind approach cannot distinguish execution paths that differ only in argument values. for example, two invocations of copy_from_user() with different size parameters hit identical basic blocks yet have vastly different security implications. We present BOUNDARY FLOW, an LLVM-based instrumentation framework that extends Linux KCOV with data-flow extraction of function arguments and return values. A compiler pass (-fsanitize-coverage=dataflow-args, dataflow-ret) emits lightweight callbacks capturing a structured tuple <PC, arg_idx, arg_size, ptr, offsets[]> at function entry and <PC, ret_size, ptr, offsets[]> at return. Composite types are automatically decomposed via DWARF DICompositeType metadata with zero source annotation. A separate kernel device(/sys/kernel/debug/kcov_dataflow) provides lock-free per-task ring buffers with no inter ference to existing KCOV or syzkaller infrastructure. We demonstrate dual utility: fuzzers gain state-aware feedback for mutation guidance into value-dependent state transitions, and security analysts obtain deterministic argument records for root-cause analysis without printk or kprobe overhead. A post-compilation pipeline (rustc, opt, llc) enables Rust kernel module instrumentation without modifying rustc, the only runtime method for capturing Rust function arguments given that drgn/vmcore fails under-O2 DWARF elision. Evaluated on five vulnerability classes (OOB, UAF, double-free, 10 deep chain propagation, Rust FFI, Rust for Linux Modules) with <3% overhead on instrumented paths.

## 1. 核心问题与研究动机

### 背景
- 现代Linux内核超过3000万行代码，并集成了Rust作为第二系统语言，其锁定层次（rwsem、mutex、RCU）创建了深度状态空间，仅靠覆盖引导的模糊测试无法完全探索。
- 现有内核模糊测试工具（如syzkaller）依赖KCOV的边覆盖（trace-pc）作为唯一反馈信号——它只报告哪些基本块被执行了，但不报告*执行时的值*。
- 边覆盖是上下文无关的：两次调用`vfs_open()`采用不同的`flags`值会命中完全相同的边，但安全影响截然不同。模糊测试器无法区分这些路径，导致在值依赖的状态转换上出现覆盖饱和。

### 核心问题
- **语义鸿沟**：KCOV的边覆盖无法捕获函数参数和返回值的具体数据流，而安全关键行为往往取决于运行时参数值而非控制流拓扑。
- 已有方案存在局限：`ftrace`/`kprobes`单函数开销高、IJON需手动标注（3000万LOC不可行）、Angora/CollAFL因多态分发而爆炸、DataFlowSanitizer开销过大（10-100×）。

### 研究目标
- 提出一个在函数边界自动捕获数据流（参数和返回值）的轻量级插桩框架，为零源标注、低开销（<3%）、同时支持C和Rust内核模块，并可直接集成到现有KCOV/syzkaller基础设施中。

## 2. 方法论：核心思想、关键技术细节与算法流程

### 核心思想
- **函数边界是自然瓶颈**：函数入口和出口既是数据流（参数）传入和传出点，又是信息密集且捕获成本低的位置。
- **在LLVM IR层自动插桩**：在现有SanitizerCoverage框架中添加新的pass，发射针对函数参数和返回值的回调，无需源级修改。

### 关键技术细节

#### 1) LLVM IR边界插桩（编译时）
- 对于每个具有`DISubprogram` debug信息的函数：
  - **参数捕获**：遍历形式参数，解析类型到`DILocalVariable` → `DIType`。若为`DICompositeType`（结构体/联合体），则自动获取其成员字段的字节偏移和大小，并计算类型名FNV-1a哈希用于去重。发射一个全局常量数组`[hash, offset0, size0, ..., offsetN, sizeN]`，并在函数入口处插入调用`__sanitizer_cov_trace_args(pc, arg_idx, arg_size, ptr, offsets_ptr, num_fields)`。标量参数被spill到alloca以统一指针接口。
  - **返回值捕获**：使用`EscapeEnumerator`定位所有`ReturnInst`，在其之前插入调用`__sanitizer_cov_trace_ret(pc, ret_size, ptr, offsets_ptr, num_fields)`。
  - 无debug信息、变参、naked函数被跳过。受两个`cl::opt`标志控制。

#### 2) 指针解引用与动态结构捕获
- 当参数是指向堆/栈/静态内存的结构体指针时，运行时回调执行一级指针解引用：**`field_addr = ptr + offset[i]`**，并通过`copy_from_kernel_nofault()`安全读取。这可以捕获KMALLOC分配的结构体内部字段，甚至能揭示释放后的KASAN毒化模式（0xfd/0xff）。
- 设计选择：仅一级解引用，不深入嵌套指针，以控制成本并避免无效指针引发的递归错误。

#### 3) 无锁每任务环形缓冲区（运行时内核后端）
- 在`task_struct`中添加三个字段：`kcov_df_area`（u64缓冲指针）、`kcov_df_size`（容量）、`kcov_df_seq`（单调递增序号）。
- **写路径**：使用`atomic_add`原子保留连续槽位；若超阈值则静默丢弃（无阻塞、无分配、无自旋锁）。指针解引用使用`copy_from_kernel_nofault()`安全处理未映射/毒化内存。`IS_ERR_VALUE`范围检测并跳过。
- **记录格式**：每记录为变长u64字序列，头3字为`[type_and_seq | pc | meta]`，随后是字段值。`type_and_seq`打包类型标记（0xE为入口，0xF为返回）与24位序号；`meta`打包8位参数索引、8位大小、48位原始指针值。
- **安全措施**：回调标记为`notrace`、`__no_sanitize_coverage`、`noinline`；在写路径顶部通过`in_task()`检查拒绝所有来自进程上下文之外的回调（softirq、hardirq、NMI），避免重入破坏。

#### 4) 独立内核设备接口
- 通过`/sys/kernel/debug/kcov_dataflow`独立于传统`/sys/kernel/debug/kcov`，使用ioctl编号空间（magic 'd'）和`mmap()`进行用户态零拷贝读取。

#### 5) Rust模块支持
- 通过后编译流水线实现：`rustc --emit=llvm-ir` → `opt`（应用自定义pass） → `llc -filetype=obj` → 标准模块链接。利用文本格式LLVM IR（.ll）避免位码版本不兼容。在`#[repr(C)]`类型的Rust结构体上自动工作（保留DICompositeType元数据）。

## 6. 主要结论与发现

1. **KCOV-DATAFLOW能在函数边界正确捕获C和Rust的参数/返回值**：在5种漏洞类上100%匹配printk真值（表1）。
2. **提供现有工具无法获取的信息**：能报告哪个具体参数被破坏、预处理值和后处理值、跨函数传播路径、以及Rust模块在-O2下的参数信息，这些在KASAN报告或drgn分析中均不可得（表2）。
3. **性能开销极低**：单回调约27 ns，平均插桩路径开销<3%（表3）；全局启用时系统调用延迟增加133%，但仍在可接受范围，且不影响非插桩路径。
4. **成功发现真实世界漏洞**：在Android Binder驱动和Rust核心模块中发现了60个测试案例，其中4个高严重性bug（如`BINDER_SET_MAX_THREADS`未验证输入导致线程耗尽拒绝服务、`Error::from_errno(-4096)`静默掩码错误码）。另有7个首次在-O2下观察到Rust行为。
5. **使函数边界契约的运行时验证成为可能**：自动检测`from_errno`预/后条件违反，无需源级插桩或tracepoint。

## 7. 优点

1. **零源标注**：完全自动的DWARF复合类型分解，无需开发者手动添加任何注释。
2. **低开销、无侵入**：单回调仅~27 ns，非插桩路径零开销；与现有KCOV和syzkaller基础设施完全兼容（独立设备接口）。
3. **Rust支持独特且唯一**：后编译流水线是目前在-O2优化下捕获Rust函数参数的唯一运行时方法（drgn/vmcore因DWARF省略而失败）。
4. **安全设计**：回调标记安全属性、使用`copy_from_kernel_nofault()`安全读取、拒绝非进程上下文回调，避免了常见陷阱（递归、锁死、崩溃）。
5. **生产可用性**：提供Kconfig选项、全局和每模块控制、权限分离（gating on kcov_df_enabled），便于集成到现有内核构建和模糊测试流水线。
6. **双重用途**：既改进了模糊测试反馈信号（状态感知突变指导），又能直接用于安全分析师进行确定性根因分析，无需printk或kprobe。

## 8. 不足与局限

1. **debug信息依赖**：完整结构体字段展开需要`CONFIG_DEBUG_INFO`和`-g`，否则仅捕获标量参数值。对于内核模糊测试这不是实际问题（KASAN已要求debug信息），但对其他场景可能有限制。
2. **Rust管道尚未集成到Kbuild**：需要手动调用包装脚本，自动化不足。作者提议未来在`rustc`中加入自带支持。
3. **缓冲区溢出静默丢弃**：当事件量超过缓冲区容量时，事件被静默丢弃。虽然这是避免死锁的有意设计，但在突发高频率调用时可能丢失关键数据。
4. **中断与NMI上下文不可见**：严格限制在进程上下文中（通过`in_task()`检查），从softirq、hardirq、NMI触发的回调被静默丢弃。因此仅通过中断触发的漏洞不可见。但这在模糊测试场景下可接受，因为模糊测试器通过系统调用运行在进程上下文中。
5. **指针解引用限于一级**：不深入嵌套指针内的指针，因此无法捕获`char *buf`的内容（只保存指针值）。对于需要跟踪深层指针内容的分析场景不够充分。
6. **缺少与syzkaller的端到端集成实验**：论文未提供定量数据（如状态覆盖提升百分比、发现新bug速度提升）来证明集成后的模糊测试效率改进，仅定性描述未来工作。
7. **实验缺乏与其他数据流方法（如Angora、CollAFL）的直接性能比较**，因为论文认为这些方法不适用于内核或开销过高；但至少可以在简化场景下给出数量级对比以加强说服力。
8. **全局启用开销在启动时间（+71%）和系统调用延迟（+133%）上较高**。作者建议连续模糊测试中采用每模块插桩，但未评估每模块插桩在实际模糊测试中的覆盖率影响。
